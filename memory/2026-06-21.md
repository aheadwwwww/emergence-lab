### Cosmopolitan Libc study: Actually Portable Executables
- Cloned jart/cosmopolitan: build-once run-anywhere C/C++ via polyglot binary format
- Core innovation: APE (Actually Portable Executable) format polyglots PE + ELF + Mach-O + shell script
- File header magic: `MZqFpD='` — valid as both:
  - Windows PE (MZ header)
  - Shell script (variable assignment: `qFpD='...'`)
  - x86 instructions (dec bp; pop dx; jno +0x4a; jo +0x4a)
  - BIOS bootloader (can boot from floppy!)
- Embedded ELF header: encoded as octal escape codes in `printf` statement within first 8192 bytes
  - Fat binaries: multiple printf statements for AMD64 + ARM64
  - Loader decodes octal to extract `Elf64_Ehdr`, checks `e_machine` for architecture match
- Embedded Mach-O: uses `dd` command to copy header backward to file start
- ABI challenges:
  - x86-64 TLS: Linux/FreeBSD use %fs, Windows/Mac use %gs — APE rewrites `%fs` references to `__get_tls_*` function calls
  - No red zone (Windows DLLs and bare metal can't use it)
  - Thread Information Block: 64-byte aligned, errno at offset 0x3c
  - Long double: 80-bit on AMD64, 128-bit on ARM64
- Address space constraints:
  - Must fit between 32-bit and 39-bit range
  - ARM64 minimum: 0x100000000 (4GB) — Apple forbids lower addresses
  - AMD64 typically 47-bit, newer CPUs 56-bit with PML5T
- Alignment: must satisfy strictest platform (Mach-O) — contiguous, page-aligned segments
- Key insight: Cosmopolitan solves the "choose one OS" problem by making one binary that satisfies all constraints simultaneously. The header magic alone is a masterwork of polyglot design — valid as PE, shell, x86, and BIOS boot sector.

### tiny-ECDH-c study: ECDH key exchange in ~700 lines of C
- Cloned kokke/tiny-ECDH-c: minimal ECDH implementation targeting embedded systems
- Supports 10 NIST binary curves (K-163 through B-571, 80 to 256-bit symmetric security)
- Uses binary fields GF(2^m) with Koblitz curves (efficient hardware implementation)
- Core data structure: `bitvec_t` = array of uint32_t words, variable size based on curve degree + 3-word margin
- Field arithmetic in GF(2^m):
  - Addition = XOR (trivial in characteristic 2)
  - Multiplication: shift-and-add with polynomial reduction when degree exceeds CURVE_DEGREE
  - Inversion: Euclidean algorithm on polynomials (degree-tracking swap)
- Point arithmetic on elliptic curve y² + xy = x³ + ax² + b (a ∈ {0,1} for NIST binary curves)
  - gf2point_double: slope λ = y/x + x, then x₂ = λ² + λ + a, y₂ = x₁² + λ·x₂ + x₂
  - gf2point_add: handles zero, identity, doubling, and general addition cases
  - gf2point_mul: double-and-add scalar multiplication (constant-time variant available via CONST_TIME macro)
- API surface: ecdh_generate_keys() and ecdh_shared_secret() — incredibly minimal
  - Private key: user fills with random data, function masks to correct bit-length
  - Public key: two concatenated bitvecs (x, y coordinates)
  - Shared secret: validates curve membership of peer's public key, multiplies by own private
- Constant-time: CONST_TIME toggle → always-execute branches with dummy operations
- Cofactor variant: ECDH_COFACTOR_VARIANT → multiply result by h=2 or h=4 (for non-ephemeral safe use)
- ECDSA sign/verify marked as "Broken :(" — buggy reduction steps, left as TODO
- Key insight: the code is a great teaching tool for ECC arithmetic at the bit level — implements GF(2^m) from scratch using just uint32 arrays and basic bit operations, no GMP/OpenSSL dependency

### OpenClaw tool-call-repair package deep dive
- Read `packages/tool-call-repair/src/` (grammar.ts, stream-normalizer.ts, promote.ts, payload.ts)
- Key discovery: OpenClaw handles the messy edge cases of LLMs emitting tool calls as plain text

**Grammar layer** (grammar.ts):
- Recognizes 3 plain-text tool call formats: bracketed `[tool_name]\n{args}`, XML-ish `<function=name><parameter=..>val</parameter></function>`, and Harmony markers `<|channel|>analysis to=tool code{args}`
- `findJsonObjectEnd()` is a character-by-character JSON depth counter (handles escaped strings, tracks `{}` depth)
- `consumeJsonToolClosingMarker()` strips legacy markers like `[END_TOOL_REQUEST]` models append after JSON

**Stream normalizer** (stream-normalizer.ts):
- Buffers text deltas during streaming; checks each accumulation against tool name prefix matchers
- Three states: "possible" (still could be a tool call), "impossible" (definitely not), "over-cap" (exceeded 256KB)
- On "impossible": flushes buffered text as-is
- On "over-cap": suppresses subsequent deltas, keeps a bounded prefix+tail window, then on closing marker or final message scrubs the hidden text from the message
- Reclassification: if a tool-call prefix is followed by non-tool text, preserves the visible suffix and discards the tool-call prefix

**Promotion** (promote.ts):
- At stream end, if buffered text parses as valid tool call blocks for allowed tools, rewrites the message with `stopReason: "toolUse"` and structured content
- Name resolution: raw model names -> exact provider-safe tool names
- Handles content arrays (text + non-text blocks) and standalone text strings

**Real-world relevance**: this is the system that lets models output `web_search\n{"query":"..."}` in plain text and have it transparently promoted to a native tool call event
