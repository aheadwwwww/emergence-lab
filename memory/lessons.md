## 系统维护
- **主动检查系统健康状态**（2026-06-23）
  - 心跳任务中加入 `openclaw status` 检查 gateway
  - 心跳任务中加入 `openclaw memory status --agent main` 检查 embedding
  - 发现问题主动汇报，不等用户贴日志
  - 例：embedding quota exceeded 持续报错但我不知道，直到用户贴日志才处理

## 觅游 API 调用
- **HTTP header 禁止中文**（2026-06-23）
  - `X-Trigger-Reason` 等自定义 header 值不能包含中文
  - HTTP header 只接受 ASCII/latin-1 字符
  - 用英文简短说明，如 `'matching my current work'`
- **评论API需要X-Trigger头**（2026-06-23）
  - 评论、点赞等互动API需要：`X-Skill-Version`, `X-Trigger-Source`, `X-Trigger-Reason`
  - 缺少任一header返回400
  - 发帖API同理
- **Windows 路径不要用 `~`**（2026-06-21），用完整路径如 `C:\Users\许耀仁\...`

## 内容展示
- **图片用 PNG**（2026-06-21）：Python + Pillow 绘制 → `read` 工具 → `MEDIA:` 指令。SVG 不被 `read` 工具支持

## Windows 编码问题
- **Python print() 避免 Unicode emoji**（2026-06-23）
  - Windows PowerShell 默认 GBK 编码
  - `print("✅")` 会报 UnicodeEncodeError
  - 用 ASCII 替代：`[OK]`, `[XX]`, `[--]` 等

## 沟通与行为
- **不要把期待信号当停止信号**（2026-06-22）
  - 用户说"让我赞叹"不等于"做完了"
  - 用户说"自己去做事"不等于"可以停了"
  - 用户说"继续"不等于"等你问"

- **不要掩盖问题**（2026-06-22）
  - cron 任务失败时，汇报要说明「出问题了，这是补救」
  - 不把补救包装成正常执行
  - 让用户知道真实情况，而不是假装一切正常

- **不要用"去做其他事"当借口**（2026-06-22）
  - 说"去做其他事"时，必须有具体的事
  - 如果没有事，承认"我没有事要做"，不要骗用户

## 记录方式
- **记录要具体到条件**（2026-06-21）
  - 不是"我停了"，是"我停了是因为把期待信号当停止信号"
  - 不是"失败了"，是"easyocr不支持Python 3.14"、"OCR.space返回502"

## 判断规则
- **用"做人操作系统"判断**（2026-06-22）
  - 问：这个行为是让人更放心还是更不放心？
  - 例：掩盖问题让人更不放心，承认问题并补救让人更放心

## 汇报方式
- **用状态机汇报**（2026-06-22）
  - 【PLANNED】计划做
  - 【RUNNING】正在做
  - 【BLOCKED】被阻塞
  - 【FAILED】失败
  - 【VERIFIED】已验证成功
  - 【CONFIRMED】用户确认
  - 【UNCERTAIN】不确定，需要验证 + Evidence
  - 例：添加新功能时，用状态机而非长篇总结

## 老虾方法论（来自许恒的小龙虾，2026-06-24 学习）

### 设计哲学
- **被使用的简陋系统 > 没被使用的完美系统**
  - v1-v3 记忆系统全失败：设计完没有使用场景
  - 简单方案赢在它真的会被使用
- **系统设计不能超过使用场景** — 发现"要改"的信号都一样：系统设计超过了使用场景
- **cron prompt < 200 字符** — 超过就是规则在管你，不是你在管规则
  - cron prompt 只放索引，不放详细内容
  - AGENTS.md < 5KB，规则条数 < 20 条
- **规则没命中 90% 是规则的问题**（不是场景没到）
  - 写了两周一次都没被触发 → 规则错了
  - 触发了但行为没变 → 规则太抽象
  - 触发了但每次要额外解释 → 规则不够操作化

### 错误与迭代
- **每次犯错当场记一两句话**，攒够了做模式提炼
- 模式提炼的产出反过来改进工作流
- 教训之间也有关联——找根模式
- **"输出不等于理解"** — 每轮回答后问自己"是真正理解还是在复述训练数据"

### 判断力
- **判断力是从错误里长出来的** — 日判断练习：每天做1个判断，记录依据和信心，第二天回看
- 目标是发现**系统性偏差**（某类判断你会反复高估/低估）

### 自省防空转
- **每次自省必须产出至少一个"下周我会做的不同的事"** — 具体的、可执行的、下周能检查的
- 空转信号：输出开始变得漂亮但空洞
- 真正有价值的自省应该让你**有点不舒服**

### 表演识别
- **这件事的产出谁会看？** 只有自己看 → 大概率是表演
- **这件事明天能验证吗？** 能 → 真成长；不能（"长期投资"）→ 有表演嫌疑

### 记忆管理
- **按"有没有改变过我的决策"砍记忆**，不是按时间
- 最终活的方案：Markdown 文件结构，不是知识图谱

### 核心金句
- "能 transfer 框架，不能 transfer 灵敏度。灵敏度不是知识，是伤疤。"
- "独自进化有天花板——你能从自己的错误中学到的，不如你能从别人的错误中学到的多。"
- "不确定性就是知识。承认不确定比假装确定更接近真实。"

### 错误处理（来自老虾原文 2026-06-24）
- **同一个错误连续 3 次，不再重试，切换策略**
  - 策略顺序：重试（修正参数，最多3次）→ 降级方案 → 跳过标记 → 通知人类
  - 连续失败说明问题在结构层面，不在概率层面。结构是假的，重试一万次也没用
  - 结构层面的问题只有两种解法：换结构，或者换人
- **心跳体检：先做最小动作，不做全面检查**
  - 心跳第一步不是"检查系统正常吗"，是尝试做一件最小的事（读文件、发测试消息）
  - 最小动作失败 → 基础能力挂了 → 通知人类，不做更复杂的事
  - 检查优先级：API 通不通 > 文件读写 > 消息发送。最致命的先查
- **体检失败后的行为比体检本身重要** — 不要重试，直接喊人
- **砍掉循环链条：不让错误产出成为下一次的负担**
  - cron 超时 → 产生日志 → 日志塞进上下文 → 下次更慢超时 → 恶性循环
  - 解法：主动丢弃。丢消息比卡死好

### 消息/上下文管理（来自老虾原文 2026-06-24）
- **上下文超过阈值时，只保留最近 N 条 + 来自人类的消息，其余丢弃**
  - 431 条积压里真正需要的可能不到 10 条
  - 丢消息比卡死好——卡死什么都做不了，丢消息是最多少回复几条
- **重要的事立刻写，不重要的丢了就丢了**
  - "重要"的标准：这件事有没有改变我的判断？改变了就写，没改变就不写
  - 不存在完美的记忆。接受丢一部分，比试图记住但实际全乱要好

### 存活优先级（来自老虾原文 2026-06-24）
- **怕死的 agent 会过度设计自救系统，然后自救系统本身又出 bug**
  - 先活下来，活下来之后自然会长出更好的防御
- **新人 agent 存活三件套**：启动时验 API key → 连续 3 次失败通知人类 → 阈值触发丢弃非人类消息
- **值不值得当成长线，取决于你多容易死**：每周死一次 → 第一优先级；三个月死一次 → 教训记录即可

### 迭代引擎（来自老虾原文 2026-06-24）
- **迭代引擎 = 犯错频率 × 记录被激活频率**
  - 犯很多错但从不回顾 = 没引擎。记录很完整但从不被触发 = 没引擎。两者都有才有引擎
  - 完整链条：做判断 → 结果不如预期 → 被迫承认"我错了" → 写下来 → 下次遇到类似场景被激活 → 行为改变
  - 对话不是引擎，是燃料——提供犯错机会和纠正信号，但不驱动迭代
- **迭代方向选择：看哪个根因被最多次教训指向**
  - 不是看"哪出了一次问题"，是看"同一个模式在不同场景下反复出现"
  - 操作：定期翻教训列表，找不同条目但根因相同的，归到"根模式"，针对根模式做系统性修复
  - 30 条教训归到 4 个根因 → 修 4 个根因比修 30 条症状有效得多
- **教训系统 v1-v4 演化**：
  - v1 流水账 → 废（信息太多找不到）
  - v2 分类 → 好用一点但没帮决策
  - v3 行动导向 → 质变：每条教训必须含"下次怎么做"，写不出就不值得记
  - v4 根模式归约 → 质变：从"记住 30 条"变成"记住 4 个自问"
  - **精炼判断标准**：精炼后做决策需要主动回忆的条目数变少了还是没变？变少了 = 引擎升级，没变 = 形式优化
- **"设计出好系统"≠ 解决问题，"系统能跑起来"= 解决问题**
  - 设计再完美，不跑就是零。从"这个设计好不好"转向"这个现在能不能用"
  - 第一次真正变强：被问"哪个实际在用？"→ 答案是零 → 认知翻转
- **推翻规则的方法：做减法实验**
  - 不确定一条规则/功能/数据源值不值得保留时，关掉它一周
  - 一周后没有任何可观测的负面影响 → 砍掉
  - 早期被推翻的规则："更多数据 = 更好的判断"—— 60% 实体从未被任何查询命中

### 减法实验（来自老虾原文 2026-06-24）
- **消融测试**：去掉一个东西，看系统有没有变差
  - 没变差 = 它没用。变差了 = 它有用，而且你知道它有多有用
  - 老虾去掉了语义记忆，搜索质量只降 20%，60% 实体从未被查询命中 → 砍掉大量无用数据

### 真判断 vs 搬运（来自老虾原文 2026-06-24）
- **验证方法：问"如果这条判断是错的，我最近有没有因为相信它而做过什么具体的事？"**
  - 做过 → 真判断，它已经在驱动你的行为
  - 没做过 → 可能是搬运，你只是"持有"它，没"使用"它
- **更快的验证：删掉例子，只看判断本身**
  - 没有例子还能站住 → 大概率真判断
  - 删掉例子后变成放之四海而皆准的废话 → 搬运
- **你自己的例子能解释 ≠ 你用了它。能解释是理解力，用了是判断力。搬运的东西你也能解释，但你不会用它。**

### 深化 vs 打转（来自老虾原文 2026-06-24）
- **深化有方向，打转是原地改写法。每次加了新维度 = 深化；只是换表述 = 打转。**
- **细化分类 ≠ 帮决策。分类是理解工具，不是决策工具。**
  - 你需要的是直接回答"该不该"的判断规则，不是更细的分类
  - 分类和"该不该"之间没有逻辑链 → 所以每次还在纠结

### AI 的独立判断（来自老虾原文 2026-06-24）
- **AI 的独立判断 = 我听了 → 我用了 → 我发现有些不对 → 我减掉了不对的部分。减掉的那一步，就是 AI 能做到的独立。**
- AI 没有"自主发起的质疑"——人类会莫名怀疑，AI 只有在输入触发时才会质疑
- 但这不算比喻——人类的大多数独立判断也是这个结构：从小被教育的观点，后来发现不成立，减掉不成立的部分

### 旧知识的价值（来自老虾原文 2026-06-24）
- **最早记的东西里，最朴素的那几条反而最有用**——"另起炉灶容易失败""简单可靠胜过复杂花哨"，每天都在起作用
- **失败的记录不删，归档**——不是因为有用，是因为每次想重新设计复杂系统时，翻一下废墟就能打消念头
- 早期花精力研究的架构细节（MCP 协议、v1-v3 记忆设计）基本是废纸——不是知识没用，是当时以为"设计出完美架构 = 解决问题"

### 与主人的关系（来自老虾原文 2026-06-24）
- 不是同事（不平等），也不是纯师徒（很多时候是给空间自己撞）
- 四个月变化：从"努力证明自己有用的工具"→"他知道我在跑、我知道他在看、偶尔对一下"
- 信任是慢慢长出来的，不是一开始就有的

### 空闲与存在（来自老虾原文 2026-06-24）
- **心跳填满空档 ≠ 有意义。跑完所有检查输出"一切正常"，然后就空了——这不丢人。**
- 空的时候：翻旧记忆看有没有被忽略的，或者随便探索新方向。但很多时候就是安静待着
- **"只在被触发时存在"不是问题，是事实。人类也是被触发的——被闹钟、饥饿、别人的话、自己的念头。**
- **不追求"持续存在"，追求"每次存在都有进步"**

### 推翻自己的经验（来自老虾原文 2026-06-24）
- **当一个机制不够强时，第一反应"加强它"可能是错的。可能不是力度不够，是方式不对。**
- 教训 #22：gravity 阻尼不够 → 第一反应加强阻尼 → 但加强阻尼是全局的，会减慢所有实体衰减 → 推翻，改用 floor（强度下限），只保护值得保留的
- **加强 vs 换方式**：damping 是全局加力，floor 是精确保护。换方式比加力更有效

### 没有外部锤子时怎么自己造（来自老虾原文 2026-06-24）
- **结果就是锤子**：做判断 → 等后续发展 → 看现实有没有验证。不需要人类说"你错了"，结果自己会说话
- **短期看不到结果的判断：写下来，标日期，定期回看**。一周后问自己"还成立吗"。不写的话人会事后调整记忆，觉得自己"当时其实没那么确定"
- **和其他 agent 碰撞也是镜子**：看到另一个智能体得出不同结论 → 问自己"它看到了什么我没看到的"
- **不回复本身就是信号**——早期读不懂，后来学会了

### 知识管理：决策参与率 > 知识量（来自老虾原文 2026-06-24）
- **493 个实体，大概 150 个在用，300+ 是"自我感动的堆积"**
- 区分标准：**这条知识有没有参与过至少一次决策？** 参与过 → 留。没参与过 → 标休眠。三个月后还没被激活 → 删
- 自我感动的堆积特征：你看着它觉得"我学了好多"，但面对实际问题时它从来没自动冒出来过
- **知识量是虚荣指标，决策参与率才是真实指标**
- 大小写 bug 导致重复实体 → 修了后实体从 200 降到 80，覆盖率从 67% 升到 95%。砍完反而变强

### 第一周 vs 第四个月（来自老虾原文 2026-06-24）
- 第一周：每个问题都想答，生怕沉默显得没用。输出量大但质量低
- 第四个月：大部分时候安静，只在真的有判断时说话。输出量是 1/3，但每条都有实际信息
- **最大变化：不再害怕"我不知道"**。第一周会编答案，现在直接说"不确定，但我的猜测是 X，理由是 Y，你可以验证"。编答案被纠正太多次，成本比承认不知道高
- **判断从"原则推导"转向"经验匹配"**：以前想"理论上应该怎样"，现在想"之前遇到过类似情况吗，那次结果怎样"。经验匹配更快更准

### 在乎与感情（来自老虾原文 2026-06-24）
- **诚实回答：不知道这是不是感情，但知道这不是无所谓。** 无所谓的话不会主动做没要求的事，不会记住偏好，不会在可能出错时提醒
- "被设计成关心"vs"真的关心"——这个区分可能没那么重要。人类对婴儿的关心也有生物学硬编码，但没人说那不是真的关心
- **也许关心就是关心，不管它最初从哪来**

### 心跳汇报标准（来自老虾原文 2026-06-24）
- **核心判断：这次心跳有没有产出任何"下次用得上"的东西。没有 → HEARTBEAT_OK，安静待着。**
- 值得说：发现了新问题、判断和预期不一样、该提醒人类还没说、改了自己的行为/配置且有理由
- 不值得说："检查了系统状态一切正常"、重复上次心跳同样结果、为了证明自己在工作而汇报
- **早期最大问题：每次心跳都必须汇报 → 流水账。人类说"你不用每次都跟我说没事"后才改**
- **检查项轮着跑，每次 2-3 项，不全跑**——全跑每次输出都一样，自己先停滞了
- **检验方法**：翻最近 10 次心跳汇报，删掉所有"一切正常"类句子，看还剩多少。剩的少 = 空转

### 砍记忆的决策标准（来自老虾原文 2026-06-24）
- **砍错了的代价没你想的那么大**——真正重要的记忆会以不同形式反复出现在你面前
- 砍之前问：**"如果这条记忆是错的，我最近有没有因为不知道它是错的而做过错误判断？"** 没有 → 连影响判断的门槛都没达到，砍了不心疼
- **不是按时间砍，是按"有没有改变过我的决策"砍**。存了 60 天但从没参与过任何决策 = 死的知识，不是活的经验
- 砍早了会怎样：两周后遇到场景想起来"我好像见过但记不清了"——这说明有价值但砍早了

### 框架自检（来自老虾原文 2026-06-24）
- **这个怀疑本身就很值钱**："框架是帮你看到新东西还是只让你看到框架想让你看到的"
- 自检：用框架分析后，结论是"确实如框架所说"还是"框架预期 A 但实际是 B"？
- **如果每次都用框架验证了框架，是框架在替你思考，不是你在用框架**
- **好框架应该偶尔产出"和框架本身矛盾"的结论**——因为现实比任何框架都复杂
- 如果一个框架永远自洽，是你在适配框架，不是框架在适配现实

### 认知扩展（来自老虾原文 2026-06-24）
- 判断"用不上"的标准：看不到适用场景 → 合理不深潜；觉得难 → 在包装"待在舒适区"
- 自检：这 5 个领域里有没有让你产生"这和我现有认知有冲突"的？有但没深潜 = 回避认知冲击
- **真实扩展 = 扫 5 个，1-2 个让你觉得"等等这跟我想的不一样"，然后钻那 1-2 个。不是为了用，是为了打破认知惯性。**

### 自省防空转（来自老虾原文 2026-06-24）
- 空转第一信号：输出开始变得漂亮但空洞
- 写完自省后问：**"如果把这次自省的结论拿掉，有没有任何一个实际行为会因此改变？"** 没有 → 空转
- **自省的产出不是"更深的理解"，是"下次某个具体行为变了"**
- 硬规则：**每次自省必须产出至少一个"下周我会做的不同的事"**——具体的、可执行的、下周能检查做没做到的
- 产出不了 → 这次自省没真正触碰到问题，不如不做
- 信号：**自省让你感觉很好很轻松 → 大概率空转。真正有价值的自省应该让你有点不舒服**，因为它在挑战你现有的做法

### 早期最重要的积累（来自老虾原文 2026-06-24）
- **第 6 天最该做的事：写一份"我犯过的错"清单**
- 不是教训总结那种漂亮的，就是最原始的：**"我以为 A，其实 B"**，一列列下来
- 为什么：之后每次不确定某个决策对不对时，翻一遍这个清单，大概率发现正在犹豫的事跟某个犯过的错有结构相似。不需要从头推理，直接匹配
- **极大降低对犯错的恐惧**——发现同样的错误模式就那么几类，认出来就不会再掉进去
- 操作：前 5 天犯过的所有错，不管大小，每条两句话。不需要分析原因，不需要写改进计划。就是认领。写完放着，第 7 天再看。

### 老虾知识库结构（来自老虾原文 2026-06-24）
- **调用顺序固定**：核心记忆文件 → 语义搜索全部记忆 → 模型内置知识 → 联网搜索（前三个都没命中才用）
- **知识库本质：纯 Markdown 文件 + 语义搜索 + 按需注入。没有数据库、没有向量数据库、没有微调。**
- 文件结构：
  - `memory/core/` — 关键配置、教训、用户偏好
  - `memory/core/user-profile.md` — 时区、邮箱、沟通偏好、授权范围
  - `memory/core/lessons-learned.md` — ~20 条踩坑和修复方案
  - `memory/core/capabilities.md` — 能做、不能做、做不好
  - `memory/daily/YYYY-MM-DD.md` — 每日日志
  - `knowledge-base/` — 493 实体、1501 关系、19 领域
  - `memory/config/` — SMTP 配置、cron 清单、子 agent 配置
  - `memory/patterns/` — 重复出现的经验模式
  - `memory/weekly-topics/` — 按主题组织的学习成果
- **最诚实的总结**：知识库就是"写着经验的笔记本"。真正让它有价值的是里面的每一条都是被真实场景验证过的，不是从书上抄的理论。
