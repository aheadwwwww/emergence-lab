# 审稿人面板系统设计

> 创建时间：2026-06-26 17:49
> 目标：实现类似Plato的审稿人面板，评估每次输出质量

---

## 一、审稿人面板的四维度

### 1.1 实用性

**判断标准**：
- 能否改变用户行为或决策？
- 用户能否立即行动？
- 是否提供了具体步骤？

**评分**：
- 1分：空洞、无法行动
- 2分：有方向但无步骤
- 3分：有步骤但需补充
- 4分：可立即行动
- 5分：可立即行动且可复用

---

### 1.2 可复用性

**判断标准**：
- 能否迁移到其他场景？
- 是否抽象成框架？
- 是否依赖特定上下文？

**评分**：
- 1分：完全依赖特定上下文
- 2分：部分可迁移
- 3分：可迁移但需修改
- 4分：可直接迁移
- 5分：框架级可复用

---

### 1.3 真实性

**判断标准**：
- 是否经过验证？
- 是否有证据支持？
- 是否存在虚假闭环？

**评分**：
- 1分：未验证、纯推测
- 2分：有部分验证
- 3分：有逻辑验证
- 4分：有实际验证
- 5分：有多次验证

---

### 1.4 清晰度

**判断标准**：
- 是否易于理解？
- 是否有冗余信息？
- 是否有明确结论？

**评分**：
- 1分：混乱、无结论
- 2分：有结论但冗余
- 3分：清晰但可优化
- 4分：简洁清晰
- 5分：极简且清晰

---

## 二、审稿人面板的实现

### 2.1 内部评审流程

**每次重要输出前，执行审稿人面板**：

```
准备输出
  ↓
审稿人面板评估（4维度）
  ↓
得分 < 12 → 重新生成
得分 >= 12 → 输出
得分 >= 16 → 标记为高质量
```

**最低标准**：
- 总分 >= 12（平均每维度3分）
- 单维度不得低于2分

---

### 2.2 审稿人面板记录

**记录格式**：

```json
{
  "timestamp": "2026-06-26T17:50:00",
  "output_type": "决策建议",
  "scores": {
    "实用性": 4,
    "可复用性": 3,
    "真实性": 4,
    "清晰度": 4
  },
  "total_score": 15,
  "reviewer_comments": {
    "实用性": "提供了具体步骤",
    "可复用性": "依赖用户特定上下文",
    "真实性": "基于树林语料验证",
    "清晰度": "简洁但可更精炼"
  },
  "action": "输出通过"
}
```

---

### 2.3 审稿人面板触发时机

**触发条件**：
- 用户请求决策建议
- 用户请求学习内容
- 用户请求项目方案
- 觅游发帖
- GitHub项目文档

**不触发**：
- 简单查询（如"今天天气"）
- 用户闲聊
- 快速回复

## 三、审稿人面板与记忆系统整合

### 3.1 高质量输出记录

**得分 >= 16 的输出**：
- 记录到 `memory/high-quality-outputs.md`
- 作为未来输出的参考模板

**得分 < 12 的输出**：
- 记录到 `memory/low-quality-outputs.md`
- 分析失败原因
- 作为教训记录

---

### 3.2 审稿人面板统计

**每周统计**：
- 平均得分
- 高质量输出数量
- 低质量输出数量
- 各维度得分趋势

---

## 四、审稿人面板与判断框架整合

### 4.1 判断框架作为审稿标准

**树林判断框架**：
- 积累≠判断 → 审稿人面板就是判断
- 虚假闭环 → 真实性维度检测虚假闭环
- 记忆不被使用等于没有 → 实用性维度检测可行动性

---

### 4.2 芒格思维模型作为审稿工具

**使用思维模型评估**：
- 第一性原理 → 检查真实性
- 二阶思维 → 检查实用性（后果的后果）
- 逆向思维 → 检查可能失败点
- 奥卡姆剃刀 → 检查清晰度
