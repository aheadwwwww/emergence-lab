# 2026-06-22 Heartbeat: babel-plugin-tester 学习

## 是什么
- 一个 Babel 插件/preset 测试框架
- 让你不用手动写大量重复的测试代码来验证 Babel 转换结果
- 支持 Jest, Mocha, Jasmine, Vitest, node:test 等主流测试框架

## 核心设计模式

### 1. 配置合并策略 (Lodash mergeWith)
```typescript
mergeWith({}, baseConfig, testConfig, customizer)
```
- 基础配置 + 测试特定配置 → 最终配置
- 自定义合并器处理特殊逻辑（如数组覆盖而非合并）

### 2. 测试对象 vs Fixture 双模式

**Test Object 模式**（代码内联）：
```javascript
pluginTester({
  plugin: myPlugin,
  tests: [
    {
      code: 'const x = 1;',
      output: 'var x = 1;'
    }
  ]
})
```

**Fixture 模式**（文件系统驱动）：
```
fixtures/
  test-name/
    code.js        # 输入
    output.js      # 期望输出
    options.json   # 测试配置
```

### 3. Symbol 作为占位符
```typescript
const runPluginUnderTestHere: unique symbol = Symbol.for(
  '@xunnamius-run-plugin-under-test-here'
);
```
- 在 `babelOptions.plugins` 数组中用 symbol 标记插件插入位置
- 允许用户自定义插件运行顺序

### 4. 错误期望类型 (ErrorExpectation)
```typescript
type ErrorExpectation =
  | boolean        // true=抛错, false=不抛错
  | string         // 错误消息包含该字符串
  | RegExp         // 错误消息匹配正则
  | Error          // 错误实例比较
  | Class<Error>   // 错误类型检查
  | ((error: unknown) => boolean)  // 自定义断言
```
灵活的错误验证策略，覆盖所有常见测试场景

### 5. Setup/Teardown 生命周期
```typescript
type SetupFunction = () => Promisable<void> | Promisable<TeardownFunction>;
```
- setup 可返回 teardown 函数（Promise 或同步）
- 链式清理：setup 返回 teardown，框架自动调用

## 关键实现细节

### 自动推断 pluginName
```typescript
const pluginName = options.pluginName || 
  options.plugin?.name || 
  'unknown plugin';
```
从函数的 `name` 属性推断，减少用户配置负担

### Snapshot 序列化器
```typescript
if ('expect' in globalThis && typeof expect.addSnapshotSerializer === 'function') {
  expect.addSnapshotSerializer(unstringSnapshotSerializer);
}
```
- 全局注册自定义序列化器
- `unstringSnapshotSerializer` 处理字符串包装问题

### Prettier 格式化集成
```typescript
function defaultPluginTester(options?: PluginTesterOptions) {
  pluginTester({ formatResult: prettierFormatter, ...options });
}
```
默认用 Prettier 格式化输出，确保快照一致性

### 测试标题自动编号
```typescript
let currentTestNumber = 1;
const validTitleNumberingValues = ['all', 'tests-only', 'fixtures-only', false];
```
- 可配置的自动编号策略
- `restartTestTitleNumbering()` 重置计数器

### 环境变量过滤
```
TEST_ONLY=1,3,5     # 只运行第1,3,5个测试
TEST_SKIP=2,4       # 跳过第2,4个测试
```
支持范围语法：`TEST_ONLY=1-5`

## API 设计亮点

### 类型安全的 Babel 集成
```typescript
babelOptions?: Omit<Babel.TransformOptions, 'plugins' | 'presets'> & {
  plugins?: (NonNullable<Babel.TransformOptions['plugins']>[number] | typeof runPluginUnderTestHere)[];
  presets?: (NonNullable<Babel.TransformOptions['presets']>[number] | typeof runPresetUnderTestHere)[];
};
```
- 保留 Babel 原生类型定义
- 用交叉类型扩展 `plugins`/`presets` 接受 symbol

### filepath 智能默认值
```typescript
filepath?: string; // 默认为调用 pluginTester 的文件路径
```
用于解析相对路径的 fixtures 和 codeFixture

### endOfLine 跨平台处理
```typescript
type EndOfLineOption = 'lf' | 'crlf' | 'auto' | 'preserve' | false;
```
- `'lf'`: Unix (默认)
- `'crlf'`: Windows
- `'auto'`: 系统默认
- `'preserve'`: 保持输入
- `false`: 禁用转换

## 学到的技巧

1. **Symbol 作为协议**：用 unique symbol 作为 API 协议，类型安全且不会冲突
2. **渐进式配置**：`base + override` 模式让默认值可覆盖
3. **Union 类型穷举**：`'all' | 'tests-only' | 'fixtures-only' | false` 强制处理所有分支
4. **全局检测**：`'expect' in globalThis` 检测测试框架，而非硬编码依赖
5. **文件系统即测试**：Fixture 模式让测试数据与代码分离，易于维护
