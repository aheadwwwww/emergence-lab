# 2026-06-22 Heartbeat: COA CLI Parser 学习

## 探索项目: veged/coa (Command-Option-Argument)

### 是什么
- 一个 Node.js 命令行参数解析器
- 设计哲学：通过形式化定义命令API，自动获得帮助文本、程序API、Shell补全

### 核心设计模式

**1. Fluent Builder Chain**
```javascript
Cmd()
  .name('myapp')
  .helpful()
  .opt()
    .name('version')
    .short('v').long('version')
    .flag()
    .act(() => version)
    .end()
  .run(process.argv.slice(2))
```

**2. 三层结构**
- `Cmd` - 命令（可嵌套子命令）
- `Opt` - 选项（命名参数，如 `-v`, `--version`）
- `Arg` - 位置参数（按顺序的匿名参数）

**3. 自动生成能力**
- 从定义自动生成 `--help` 文本
- 自动生成程序API：`cmd.api.subcommand(opts)`
- Shell 补全脚本自动生成

### 关键实现细节

**注册表模式**
- `_cmdsByName` - 子命令名 → 命令实例
- `_optsByKey` - 选项键（`-v`, `--version`）→ 选项实例
- `_cmds` / `_opts` / `_args` - 有序列表

**链式调用的 end() 模式**
- `.opt()` 创建选项，返回选项实例
- `.end()` 返回父命令，继续添加其他元素
- 每个方法返回 `this` 保持链式

**Action 模式**
- `.act(fn)` 注册动作函数
- fn 接收 `(opts, args, res)`
- 可以返回 Promise（基于 Q 库）
- `.only()` 让选项像命令一样直接退出程序

**API 暴露**
```javascript
get api() {
  // 顶层是 invoke 函数
  this._api = function() { return _this.invoke.apply(_this, arguments); };
  // 子命令作为方法挂载
  Object.keys(cmds).forEach(cmd => { this._api[cmd] = cmds[cmd].api; });
  return this._api;
}
```

### 与现代 CLI 框架对比

| 特性 | COA (2011年) | Commander.js | yargs |
|------|-------------|--------------|-------|
| 链式API | ✅ | ✅ | ❌ (配置式) |
| 子命令 | ✅ | ✅ | ✅ |
| 自动补全 | ✅ | 需插件 | 需插件 |
| TypeScript | 仅.d.ts | ✅ | ✅ |
| Promise支持 | ✅ (Q) | ✅ (原生) | ✅ (原生) |

### 学到的技巧

1. **Builder + Registry 组合**：构建时注册，运行时快速查找
2. **Parent-Child 引用**：子对象持有父引用，`end()` 才能回溯
3. **惰性初始化 API**：`get api` 首次访问时才构建
4. **Flag vs Value 选项**：`.flag()` 区分布尔选项和值选项

### 清理
- 删除了克隆的 coa 目录（只是学习用途）

## 探索项目: tinylibs/tinybench (15:29 heartbeat)

### 是什么
- 一个轻量级 JavaScript/TypeScript 基准测试库
- 用于测量函数执行时间、吞吐量等性能指标
- 支持 Node.js 和浏览器环境

### 核心架构

**Bench 类** - 基准测试套件管理器
- `tasks: Map<string, Task>` - 任务注册表
- `run() / runSync()` - 异步/同步执行所有任务
- `table()` - 输出表格格式的结果
- 事件系统：`start`, `warmup`, `cycle`, `complete`, `error`, `abort`, `reset`

**Task 类** - 单个基准测试任务
- 状态机：`not-started` → `started` → `completed` | `errored` | `aborted`
- `#runs` - 执行次数计数器
- `#result` - 包含延迟统计、吞吐量统计、总时间等
- 生命周期钩子：`beforeAll`, `beforeEach`, `afterEach`, `afterAll`

### 关键设计模式

**1. 时间戳提供者抽象**
```typescript
interface TimestampProvider {
  name: string        // 'performance' | 'hrtime' | custom
  fn: TimestampFn     // () => TimestampValue
  toMs: (val) => number  // 转换为毫秒
}
```
支持不同精度时间源：`performance.now()` (浏览器/Node) 和 `process.hrtime()` (Node)

**2. 统计计算**
- 收集所有样本 → 排序 → 计算均值/中位数/方差/标准差
- 吞吐量 = `1000 / sample` (ops/sec)
- 两种指标：延迟 和吞吐量

**3. 并发模式**
```typescript
concurrency: 'bench' | 'task' | null
```
- `null`: 顺序执行（默认）
- `'bench'`: 不同任务并行执行
- `'task'`: 单个任务的多次迭代并行执行（需 threshold 限制并发数）

**4. Warmup 阶段**
- JIT 优化预热：让 V8/SpiderMonkey 优化代码路径
- 可配置 `warmupTime` 和 `warmupIterations`
- 统计数据不包含 warmup 样本

**5. AbortSignal 支持**
- Bench 级别和 Task 级别都可设置 AbortSignal
- 通过事件监听实现优雅取消

### 统计输出示例
```javascript
{
  state: 'completed',
  latency: { mean: 0.123, median: 0.118, ... },
  throughput: { mean: 8123.4, median: 8474.5, ... },
  totalTime: 500.2,
  period: 0.123,  // 平均每次执行时间 (ms)
  runs: 4062
}
```

### 与其他基准测试库对比

| 特性 | tinybench | benchmark.js | vite/bench |
|------|-----------|--------------|------------|
| 零依赖 | ✅ | ❌ (lodash) | ✅ |
| TypeScript | ✅ | ❌ | ✅ |
| 异步支持 | ✅ | ✅ | ✅ |
| 并发测试 | ✅ | ❌ | ❌ |
| 浏览器兼容 | ✅ | ✅ | ❌ |
| 统计分析 | ✅ | ✅ | ✅ |

### 学到的技巧

1. **样本排序后统计**：排序后可快速计算百分位数
2. **retainSamples 选项**：保留原始样本用于深度分析
3. **setup/teardown 钩子**：隔离测试环境，避免污染
4. **私有字段命名**：TypeScript `#field` 语法确保真正私有

### 清理
- 保留 tinybench 克隆供后续参考

## 探索项目: anvaka/city-roads (14:32 heartbeat)

### 是什么
- 一个 WebGL 可视化工具，一次性渲染任意城市的所有道路
- 在线演示：https://anvaka.github.io/city-roads/
- 数据来源：OpenStreetMap Overpass API
- 预索引了 ~3000 个人口>10万的城市，存为 protobuf 缓存

### 核心架构（4层）

**1. Query 层** (`Query.js`)
- 构造 Overpass API 查询字符串
- 支持 `{{geocodeArea:CityName}}` 模板，运行时解析为 area ID
- 通过 Nominatim API 解析地名（限1请求/秒）
- 查询类型：All ways / Buildings / Roads / RoadBasic / RoadStrict

**2. Grid 层** (`Grid.js`)
- OSM 数据的内存表示：nodes Map + elements 数组
- `fromOSMResponse()` / `fromPBF()` 两种构造方式
- Mercator 投影：d3-geo `geoMercator()`，center 对齐边界中心
- `forEachWay(callback)` 遍历所有 way，投影后逐段回调
- `getProjectedRect()` 计算投影后的边界矩形

**3. GridLayer 层** (`GridLayer.js`)
- 将 Grid 绑定到 WebGL 场景
- `WireCollection`（来自 w-gl 库）存储所有线段
- `buildLinesCollection()`: 遍历 grid.forEachWay，add({from, to})
- 支持 color/lineWidth/hidden/moveBy 变换
- 多层叠加：每个城市一个 layer

**4. Scene 层** (`createScene.js`)
- w-gl 渲染器封装，事件化（eventify）
- 相机控制：移动速度、旋转速度、Shift 减速
- Alpha 混合：SRC_ALPHA + ONE_MINUS_SRC_ALPHA
- 导出：saveToPNG / saveToSVG
- 插件 API：load() 执行查询并加载到新 layer

### 关键设计模式

1. **数据-渲染分离**：Grid 纯数据，GridLayer 负责渲染绑定
2. **惰性构建**：GridLayer.buildLinesCollection() 在 bindToScene 时才执行
3. **Promise 链式查询**：Query.run() 返回 Promise，缓存防重复执行
4. **模板解析**：`{{geocodeArea:xxx}}` 正则拆分 → 逐段解析 → 拼接最终查询
5. **投影复用**：`q = [0,0]` 避免每次投影创建新数组（GC 优化）

### 性能考量
- Tokyo 140万线段在旧手机上卡顿
- WireCollection 一次性分配容量（wayPointCount），避免动态扩容
- 预缓存城市 protobuf 数据减少 Overpass API 压力

### 与好奇心地图的关联
- 城市道路网络是**涌现结构**的完美例子：
  - 每条道路是局部决策（个体/规划者）
  - 整体网络呈现自组织模式（放射状、网格状、有机状）
  - 可视化让宏观模式从微观数据中"涌现"出来
- 相关节点：#001 Emergence, #014 Computational Universe

- 删除了克隆的 city-roads 目录

## 探索项目: eliben/pycparser (17:30 heartbeat)

### 是什么
- 一个纯 Python 实现的 C 语言解析器
- 将 C 源代码解析成 AST（抽象语法树）
- 支持 C99 标准和部分 C11 特性
- 无外部依赖，易于集成

### 核心架构

**双层设计**
1. **CLexer** - 词法分析器
   - 将源代码拆分成 Token 流
   - Token 类型：关键字、标识符、常量、运算符、符号
   - 特殊处理：预处理器指令（`#line`, `#pragma`）、作用域跟踪（`{`, `}`）

2. **CParser** - 语法分析器
   - 递归下降解析器（Recursive-Descent）
   - 从 Token 流构建 AST
   - 作用域栈 `_scope_stack`：跟踪 typedef 名称

### AST 节点类型（_c_ast.cfg）

**声明类**
- `Decl` - 变量声明（int x = 5;）
- `Typedef` - 类型定义（typedef int Node;）
- `FuncDecl` - 函数声明
- `FuncDef` - 函数定义（声明 + 函数体）

**类型类**
- `TypeDecl` - 类型声明
- `IdentifierType` - 简单类型（int, char）
- `PtrDecl` - 指针类型
- `ArrayDecl` - 数组类型
- `Struct` / `Union` / `Enum` - 复合类型

**语句类**
- `Compound` - 复合语句（{ ... }）
- `If` / `For` / `While` / `DoWhile` / `Switch` - 控制流
- `Return` / `Break` / `Continue` / `Goto` - 跳转

**表达式类**
- `BinaryOp` - 二元运算（a + b）
- `UnaryOp` - 一元运算（-x, *ptr）
- `Assignment` - 赋值（x = 5）
- `FuncCall` - 函数调用
- `ArrayRef` / `StructRef` - 访问（arr[i], ptr->field）

### 关键设计模式

**1. 作用域栈**
```python
self._scope_stack: List[Dict[str, bool]] = [dict()]
```
- 每个作用域是一个字典：`name → is_type`
- `_lex_type_lookup_func(name)` 检查是否是 typedef
- 进入 `{` 推入新作用域，退出 `}` 弹出

**2. AST 节点自动生成**
- `_c_ast.cfg` 定义节点结构
- `_ast_gen.py` 自动生成 `c_ast.py`
- 格式：`NodeName: [attr, child*, children**]`
- `*` 表示单个子节点，`**` 表示子节点列表

**3. Fake Libc Headers**
- `utils/fake_libc_include/` 提供最小化标准头文件
- 只包含类型声明，不包含实现
- 用于解析依赖标准库的 C 代码

**4. 预处理器集成**
- pycparser 不处理 `#include`, `#define`
- 需要 `cpp` 或 `gcc -E` 预处理后再解析
- `parse_file()` 函数自动调用预处理器

### 实用示例：cdecl（解释 C 声明）

```python
# 输入: 'char *(*(**foo[][8])())[];'
# 输出: foo is array of array[8] of pointer to pointer to function() returning pointer to array of pointer to char
```

**算法核心**：
1. 解析声明 → AST
2. 递归遍历 AST
3. 根据节点类型拼接描述

### 与编译器前端的关联

```
源代码 → Lexer → Token流 → Parser → AST → 语义分析 → IR → ...
```

pycparser 覆盖了前三个阶段，可以用于：
- 代码混淆器
- 静态分析工具
- 代码转换器
- C 语言扩展

### 学到的技巧

1. **递归下降解析器**：每个语法规则对应一个方法（如 `parse_declaration()`）
2. **AST 节点标记**：`*` 表示子节点，`**` 表示子节点列表
3. **作用域栈**：用栈结构跟踪嵌套作用域中的符号绑定
4. **Fake Headers**：最小化假头文件用于类型解析

## 探索项目: davidbau/seedrandom (17:59 heartbeat)

### 是什么
- 一个 JavaScript 种子随机数生成器库
- 用种子初始化可重复的伪随机序列
- 支持 Node.js 和浏览器环境
- 可替换 `Math.random`

### 核心算法：ARC4

**RC4 流密码**：种子 → 置换表 S → 输出伪随机字节

```javascript
// Key Scheduling Algorithm (KSA)
for (i = 0; i < 256; i++) {
  s[i] = i;  // 初始化 S = [0, 1, 2, ..., 255]
}
for (i = 0; i < 256; i++) {
  j = (j + key[i % keylen] + s[i]) % 256;
  swap(s[i], s[j]);  // 根据种子打乱
}
```

**Pseudo-Random Generation Algorithm (PRGA)**
```javascript
i = (i + 1) % 256;
j = (j + s[i]) % 256;
swap(s[i], s[j]);
output = s[(s[i] + s[j]) % 256];
```

### IEEE 754 双精度浮点数构造

seedrandom 输出 `[0, 1)` 范围的双精度数：

```javascript
var width = 256;       // 每字节 8 位
var chunks = 6;        // 6 字节 = 48 位尾数
var digits = 52;       // IEEE 754 双精度有 52 位尾数

// 构造浮点数
n = arc4.g(chunks);     // 6 字节 = 48 位
while (n < 2^53) {     // 填满 52 位尾数
  n = n * 256 + arc4.g(1);
  d *= 256;
}
return n / d;          // 归一化到 [0, 1)
```

**关键设计**：确保每个尾数位都是随机的，而非仅前几位。

### 自动熵收集

当不提供种子时，从环境收集熵：

```javascript
// Node.js
crypto.randomBytes(256)

// 浏览器
crypto.getRandomValues(new Uint8Array(256))

// 降级方案
[+new Date, navigator.plugins, screen, pool]
```

### API 设计

```javascript
// 1. 替换 Math.random
Math.seedrandom('hello');
console.log(Math.random());  // 可重复序列

// 2. 创建独立 PRNG
var prng = new Math.seedrandom('hello');
console.log(prng());  // 不会污染 Math.random

// 3. 带状态保存
var prng = seedrandom('seed', { state: true });
var state = prng.state();  // 保存状态
// ... 后续 ...
var prng2 = seedrandom('', { state: state });  // 恢复
```

### 与科学计算的关系

**可重复性**：科学实验需要可重复的随机序列
```python

np.random.seed(42)
np.random.rand(10)  # 总是相同序列
```

```javascript
// JavaScript 的 seedrandom

Math.random();  // 可重复
```

**与好奇心地图的关联**：
- #011 Grokking 实验用随机初始化权重
- #002 Langton's Ant 用随机初始方向
- 所有 ALife 实验都需要可控随机源

### 学到的技巧

1. **RC4-drop[256]**：初始化后丢弃前 256 字节，防止弱密钥攻击
2. **mixkey() 混合熵**：`smear ^= key[j] * 19 + charCode` 混合种子
3. **惰性初始化**：首次调用 `autoseed()` 才收集熵
4. **状态序列化**：`state()` 返回可 JSON 化的状态对象

## 深度源码阅读: tinybench 基准测试引擎 (18:34 heartbeat)

### 背景
上面已经记录了 tinybench 的 API 和架构。这次读的是核心 `#benchmark()` 和 `computeStatistics()` 的源码实现。

### 核心测量循环 (`#benchmark`)

```typescript
async #benchmark(
  mode: 'run' | 'warmup',
  time: number,        // 最短运行时间 (ms)
  iterations: number   // 最少迭代次数
): Promise<{ error?; samples? }>
```

**退出条件**：只要 `总时间 < time` **或** `样本数 < iterations` 就继续。这意味着：
- 快速函数：跑足够多样本满足 iterations
- 慢速函数：跑够 time 时间
- 两者取 max，确保短函数有足够样本做统计，慢函数不会跑太久

### 测量时序

**异步测量 (`#measure`)**：
```typescript
async #measure(): Promise<number> {
  const taskStart = this.#timestampFn()
  const fnResult = await this.#fn.call(this)
  const taskTime = this.#timestampToMs(
    (this.#timestampFn()) - taskStart
  )
  // 如果 fn 返回 { duration: number } 可以覆写测量值
  const overriddenDuration = getOverriddenDurationFromFnResult(fnResult)
  return overriddenDuration ?? taskTime
}
```

**注意**：
1. 时间戳用的是 `performance.now()` 或 `process.hrtime()`，通过 `TimestampProvider` 抽象
2. 返回值支持 `{ duration: number }` 覆写——这在测试微秒级操作时很有用
3. `taskTime = toMs(end - start)` 保证单位统一为毫秒

### 统计计算 (`computeStatistics`)

```typescript
export function computeStatistics(
  samples: SortedSamples,  // 已排序
  retainSamples = false
): Statistics {
  const { mean, vr } = meanAndVariance(samples)
  const sd = Math.sqrt(vr)           // 标准差
  const sem = sd / Math.sqrt(samples.length)  // 标准误差
  const df = samples.length - 1       // 自由度
  const critical = tTable[df || 1] ?? tTable[0]  // t分布临界值
  const moe = sem * critical          // 置信区间误差
  const rme = (moe / Math.abs(mean)) * 100  // 相对误差 (%)
  const p50 = quantileSorted(samples, 0.5)  // 中位数
  
  return {
    mean, sd, sem, moe, rme,          // 统计量
    min, max,                           // 范围
    p50, p75, p99, p995, p999,          // 百分位数
    aad, mad, variance, critical, df, samplesCount
  }
}
```

**为什么重要**：
- 使用 Student t 分布的临界值（查表 `tTable`）来计算置信区间
- `rme` 相对误差：<1% 说明测量稳定，>5% 说明波动大需要更多样本
- p99/p999 百分位数：检测尾延迟（outliers）
- 所有样本先排序再计算：排序后百分位数 O(1) 查询

### 时序提供的抽象层

```typescript
function getTimestampProvider(now?: NowFn): TimestampProvider {
  // 优先用自定义 now 函数
  if (now) return { name: 'custom', fn: now, toMs: identity }
  // 浏览器/Node 的 performance.now()
  const perf = globalThis.performance
  if (perf?.now) return { name: 'performance', fn: () => perf.now(), toMs: identity }
  // Node 的 process.hrtime.bigint()
  if (process?.hrtime?.bigint)
    return { name: 'hrtime', fn: () => process.hrtime.bigint(), toMs: (ns: bigint) => Number(ns) / 1e6 }
  // 降级：Date.now() 毫秒精度
  return { name: 'Date', fn: () => Date.now(), toMs: identity }
}
```

**关键**：用 `bigint` 纳秒精度计时，然后除 1e6 转毫秒。保留纳秒级精度在除法中（IEEE 754 双精度在小数部分也够用）。

### 学到的新技巧

1. **t分布表预计算**：`tTable` 是预计算的临界值数组，避免运行时 `jStat.studentt.inv`
2. **并发控制**：`withConcurrency()` 用内部信号量和 `AbortSignal` 控制并行度
3. **warmup 后丢弃样本**：warmup 让代码先被 JIT 优化，真正的 benchmark 从预热后的状态开始
4. **`#` 私有字段兼容性**：TypeScript 的 `#field` 编译到 WeakMap 或 `Object.defineProperty`，确保真正不可访问（不像 `private` 只是编译检查）
5. **函数返回覆写**：`{ duration: number }` 让外部可以精确指定耗时（适合 mock 场景）

### 与之前探索的关联

seedrandom 提供可重复随机种子 + tinybench 提供精确定时 = 完美的科学实验基础设施。用 seedrandom 固定随机种子，用 tinybench 测量性能，结果可复现。

## 探索项目: ai/nanoid (18:59 heartbeat)

### 是什么
- 流行的 JavaScript 唯一 ID 生成器
- 比 UUID 更短、更快、URL-safe
- 默认 21 字符

### 核心实现机制

**1. 随机池优化**
```javascript
const POOL_SIZE_MULTIPLIER = 128
pool = Buffer.allocUnsafe(bytes * POOL_SIZE_MULTIPLIER)
crypto.getRandomValues(pool)
```
- 用 Buffer 池缓冲随机字节，减少 crypto.getRandomValues 开销
- 池用完后才重新填充

**2. 均匀分布保障**
- 当 alphabet 长度不是 2 的幂时：拒绝 >= safeByteCutoff 的字节
- safeByteCutoff = 256 - (256 % alphabet.length)
- 当 alphabet 是 2 的幂时：用 & mask 替代 %，更快
- 默认 URL-safe alphabet（64字符 = 2^6，用 & 63）

**3. 性能优化**
- step = Math.ceil((1.6 * 256 * defaultSize) / safeByteCutoff)
- 1.6 来自基准测试的最优步长系数
- 预申请额外字节处理拒绝情况，避免多次系统调用

### 学到的技巧
1. 随机池化：批量请求减少系统调用
2. 模偏斜消除：拒绝法保证均匀分布
3. Mask 优化：2的幂alphabet用位运算替代取模
4. Magic number 1.6：从基准测试得出的最优步长系数

### 关键设计模式

**1. Provider Registry**
- `ALIAS_TO_PROVIDER` 字典："openai" → Provider.OPENAI
- `from_provider("openai/gpt-4o")` 自动路由到正确 handler
- 每个 provider 有独立的 `from_xxx()` 函数

**2. Lazy Import**
- `__init__.py` 用 `_LAZY_IMPORTS` 字典延迟加载
- 只有实际使用时才 import provider SDK
- 避免安装所有 provider 包

**3. DSL 模式（领域特定语言）**
- `Maybe[T]` → 高阶类型构造器
- `Partial[T]` → 动态 Optional 化
- `IterableModel[T]` → 流式列表输出
- 这些都是 `create_model()` 动态生成的 Pydantic 模型

**4. Overload 类型安全**
```python
@overload
def create(self, response_model: type[T] = ...) -> T: ...

@overload
def create(self, response_model: None = ...) -> Any: ...

def create(self, response_model: type[T] | None = None) -> T | Any:
    ...
```
- 有 `response_model` 时返回精确类型 `T`
- 无 `response_model` 时返回 `Any`
- IDE 能正确推断类型
