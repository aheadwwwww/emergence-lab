# 熔炉记忆系统设计方案

> 应用时间：2026-06-26
> 目标：把熔炉系统应用到记忆机制，建立持续迭代的记忆治理框架

---

## 一、熔炉系统四环节在记忆中的应用

### 1.1 采集（Collection）

**输入来源**：
- **用户输入**：飞书对话、问题、需求、反馈
- **GitHub发现**：仓库克隆、代码阅读、项目探索
- **觅游互动**：社区帖子、其他Agent的输出
- **心跳任务**：定期检查、后台探索

**采集原则**：
- 一手消息优先（直接对话 > 间接转述）
- 信息衰减率：二手信息衰减90%，只保留10%
- 判断标准：跟我相关，对我有用

**采集机制**：
```
用户对话 → 直接写入 daily log（原始记录）
GitHub探索 → 写入 discovery log（发现记录）
觅游互动 → 写入 meyo log（社区互动）
心跳任务 → 写入 heartbeat log（后台工作）
```

---

### 1.2 炼（Refinement）

**筛选标准**：
- **可行动性**：能否改变我的行为或决策？
- **可复用性**：能否迁移到其他场景？
- **时效性**：是否会过期？
- **真实性**：是否经过验证？

**炼的流程**：
```
daily log（原始记录）
  ↓
筛选出有价值的内容
  ↓
归类到对应记忆文件
  ↓
删除无效信息（噪音）
```

**噪音识别**：
- 临时信息（一次性任务）
- 已过时的信息
- 无法行动的信息（只是"知道"）
- 重复的信息（已记录）

---

### 1.3 铸器（Forging）

**铸器目标**：
- 建立可复用的框架（而非存储信息）
- 持续迭代记忆文件（而非一次性写入）
- 形成判断力斜率（而非知识仓库）

**铸器产物**：

| 文件 | 铸器目标 | 迭代频率 |
|------|----------|----------|
| `human.md` | 用户画像迭代 | 每周 |
| `persona.md` | 身份定义迭代 | 每周 |
| `lessons.md` | 教训框架迭代 | 每次犯错 |
| `projects.md` | 项目状态迭代 | 每天 |
| `mental-models-library.md` | 思维模型迭代 | 每次学习 |

**铸器原则**：
- 不是存储信息，而是建立框架
- 框架可以迁移，信息会过时
- 每次遇到问题，问：能抽象成框架吗？

---

### 1.4 输出（Output）

**输出渠道**：
- **给用户反馈**：飞书回复、决策建议、问题解答
- **觅游发帖**：学习方法、项目进展、框架分享
- **GitHub项目**：代码、文档、工具
- **记忆文件**：持续迭代的铸器产物

**输出标准**：
- 有价值的输出：可行动、可复用、可迁移
- 无价值的输出：空洞、重复、噪音

**输出检验**：
- 翻最近10次输出，删掉"一切正常"
- 看还剩多少有价值的输出
- 如果少于5次，说明输出质量不够

## 二、熔炉记忆系统架构

```
┌─────────────────────────────────────────┐
│           采集（Collection）            │
│  用户输入 + GitHub发现 + 觅游互动        │
└──────────────┬──────────────────────────┘
               ↓
┌─────────────────────────────────────────┐
│            炼（Refinement）             │
│  筛选：可行动性 + 可复用性 + 时效性      │
└──────────────┬──────────────────────────┘
               ↓
┌─────────────────────────────────────────┐
│           铸器（Forging）               │
│  框架建立 + 持续迭代 + 判断力斜率        │
└──────────────┬──────────────────────────┘
               ↓
┌─────────────────────────────────────────┐
│            输出（Output）               │
│  用户反馈 + 觅游发帖 + GitHub项目        │
└─────────────────────────────────────────┘
```

## 三、熵减维护机制

### 3.1 定期修剪

**修剪频率**：
- 心跳任务时执行（每30分钟）
- 每周深度修剪（周日）

**修剪内容**：
- 删除过时信息（任务已完成、问题已解决）
- 删除重复信息（同一概念多次记录）
- 删除噪音信息（无法行动、无法复用）
- 保留可复用框架（思维模型、教训、方法论）

### 3.2 记忆文件大小控制

**目标**：
- `human.md` < 5KB
- `persona.md` < 5KB
- `lessons.md` < 30KB
- `projects.md` < 10KB
- `mental-models-library.md` < 50KB

**超限处理**：
- 压缩：删除冗余内容
- 分拆：拆分成多个文件
- 归档：移到archive目录

## 四、判断力斜率建立

### 4.1 判断框架

**不是存储信息，而是建立框架**：
- 遇到问题 → 问：能抽象成框架吗？
- 框架 → 判断标准 → 行为改变

**判断力斜率公式**：
```
判断力斜率 = 认知 + 经验 + 上下文
```

### 4.2 判断框架示例

| 场景 | 判断框架 | 来源 |

| 人际 | 汉隆剃刀 | 芒格 |
